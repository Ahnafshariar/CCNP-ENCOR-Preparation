# OSPF — LSA Types, Area Types & How They Relate

*From Class 12 (18 Jul 2026) and Class 13 (20 Jul 2026)*

---

## The Core Relationship

LSA types describe **what information flows** through OSPF. Area types control **which LSAs are allowed in.** Understanding both — and how they map to each other — is the key to mastering OSPF.

---

## Part 1: LSA Types

Every piece of routing information in OSPF is carried by a **Link State Advertisement (LSA).** Each type serves a different purpose and is generated by a different router role.

### Type 1 — Router LSA

| | |
|---|---|
| **Generated by** | Every OSPF router |
| **Scope** | Stays within the area (never crosses an ABR) |
| **Contains** | All of the router's links (interfaces) and their costs within that area |
| **Seen on** | Point-to-point links, all network types |
| **Show command** | `show ip ospf database router` |

Every router generates exactly one Type 1 LSA per area it belongs to. On a P2P link, Type 1 is the only LSA you'll see (no DR = no Type 2). This is the foundation of the LSDB — without Type 1, OSPF can't build the topology map.

```
R1# show ip ospf database router
  ! Lists every link this router has in the area
  ! Link Type: point-to-point, stub, transit
```

### Type 2 — Network LSA

| | |
|---|---|
| **Generated by** | The **DR** (Designated Router) only |
| **Scope** | Stays within the area |
| **Contains** | List of all routers on that multi-access segment |
| **When it exists** | Only on **broadcast** (or NBMA) segments — never on P2P |
| **Show command** | `show ip ospf database network` |

If there's no DR (P2P link), there's no Type 2 LSA. On my Lab 07 backbone (10.0.0.0/24 via the switch), R2 is the DR and generates the Type 2 LSA listing R1, R2, R3, R4 on that segment.

**Key exam point:** P2P links have Type 1 only. Broadcast links have Type 1 + Type 2.

### Type 3 — Summary LSA (Inter-Area)

| | |
|---|---|
| **Generated by** | **ABR** (Area Border Router) |
| **Scope** | Crosses area boundaries (that's the whole point) |
| **Contains** | Prefixes from one area, advertised into another |
| **Route table marker** | `O IA` (OSPF Inter-Area) |
| **Show command** | `show ip ospf database summary` |

The ABR takes its knowledge of area 26's networks and creates Type 3 LSAs to advertise them into area 0. It does NOT forward the original Type 1/2 LSAs — it creates fresh Type 3 summaries. This is how areas stay independent.

```
! On R1 (area 0), seeing area 26's networks:
R1# show ip route ospf
  O IA  10.2.6.0/24     ← this came from a Type 3 LSA generated by R2 (ABR)
  O IA  192.168.100.6   ← R6's loopback, also Type 3 from R2
```

**What blocks Type 3:** Totally Stubby and Totally NSSA areas block Type 3 at the ABR. The ABR injects only a default route instead.

### Type 4 — ASBR Summary LSA

| | |
|---|---|
| **Generated by** | **ABR** (NOT the ASBR itself) |
| **Scope** | Crosses area boundaries |
| **Contains** | "How to reach the ASBR" — a pointer, not route details |
| **Purpose** | Tells routers in other areas where the ASBR is, so they can follow Type 5 routes |
| **Show command** | `show ip ospf database asbr-summary` |

This is the most confusing LSA. Here's why it exists: R1 is an ASBR in area 0, redistributing external routes (Type 5). R6 is in area 26 and receives those Type 5 LSAs. But R6 doesn't know where R1 is — it's in a different area. Type 4 LSA tells R6: "The ASBR that generated those Type 5 routes is reachable via R2 (my ABR)."

Without Type 4, Type 5 is useless — you know the destination but not how to reach the router that knows the path.

**What blocks Type 4:** Stub areas block Type 4 (along with Type 5). That's why stub areas get a default route instead — they don't need to know where the ASBR is.

### Type 5 — AS External LSA

| | |
|---|---|
| **Generated by** | **ASBR** (the router doing redistribution) |
| **Scope** | Floods the **entire OSPF domain** (all areas, unless blocked) |
| **Contains** | Routes redistributed from outside OSPF (BGP, EIGRP, static, another OSPF process) |
| **Route table marker** | `O E1` or `O E2` |
| **Default metric type** | **E2** (metric does not increment as it crosses routers) |
| **Show command** | `show ip ospf database external` |

In my Lab 07, R1 redistributes OSPF 100 (R7's routes) into OSPF 10 as Type 5. These flood everywhere except stub and NSSA areas.

**E1 vs E2:**
- **E2 (default):** metric stays fixed no matter how many hops away. If the ASBR sets metric 10000, every router sees 10000.
- **E1:** metric increments as it crosses each router (adds the internal OSPF cost). More accurate but less common.

**What blocks Type 5:** Stub, Totally Stubby, NSSA, Totally NSSA — all four block Type 5 at the ABR.

### Type 7 — NSSA External LSA

| | |
|---|---|
| **Generated by** | **ASBR inside an NSSA area** |
| **Scope** | Only within the NSSA area |
| **Contains** | Redistributed external routes (same info as Type 5, different LSA type) |
| **Route table marker** | `O N1` or `O N2` |
| **Conversion** | ABR converts Type 7 → Type 5 at the area boundary |
| **Show command** | `show ip ospf database nssa-external` |

Type 7 exists because NSSA blocks Type 5 — so external routes can't enter as Type 5. But the NSSA still needs to inject its OWN external routes (e.g., R5 redistributing OSPF 900 in area 35). Type 7 is the workaround: it carries external routes inside the NSSA, and the ABR (R3) converts them to Type 5 before flooding into area 0.

## Part 2: Area Types

Each area type is defined by **which LSA types it allows through the ABR.**

### Standard (Normal) Area
- Accepts **everything**: Type 1, 2, 3, 4, 5
- No filtering at the ABR
- Used in: my Lab 06 (all areas), Lab 07 area 0 and area 48

### Stub Area
- **Blocks:** Type 4 and Type 5 (no external routes, no ASBR pointers)
- **Replaces with:** default route (`O*IA 0.0.0.0/0`) injected by ABR
- **Keeps:** Type 1, 2, 3 (intra-area and inter-area routes)
- **No redistribution allowed** inside the area
- Used in: Lab 07 area 26 (R2 ABR, R6 internal)
- Config: `area 26 stub` on ALL routers in the area

### Totally Stubby Area
- **Blocks:** Type 3, 4, and 5 (no inter-area, no externals, no ASBR pointers)
- **Replaces with:** default route only
- **Keeps:** Type 1, 2 (intra-area routes only)
- **No redistribution allowed** inside the area
- Used in: Lab 07 area 411 (R4 ABR, R11 internal)
- Config: `area 411 stub no-summary` on the **ABR only**; `area 411 stub` on internal routers

### NSSA (Not-So-Stubby Area)
- **Blocks:** Type 5 (no external routes from outside the area)
- **Allows:** Type 7 (local redistribution within the NSSA)
- **Keeps:** Type 1, 2, 3
- **ABR converts** Type 7 → Type 5 at the area boundary
- Used in: Lab 07 area 35 (R3 ABR, R5 ASBR redistributing OSPF 900)
- Config: `area 35 nssa` on ALL routers in the area

### Totally NSSA
- **Blocks:** Type 3, 4, and 5
- **Allows:** Type 7 (local redistribution)
- **Replaces inter-area and external with:** default route only
- Config: `area X nssa no-summary` on the **ABR**; `area X nssa` on internal routers
- Note: not used in Lab 07 but covered in Class 13 notes

### Backbone (Area 0)
- **Always Normal.** Cannot be stub, NSSA, or any other type.
- Every area must connect to it (directly or via virtual-link).
- All inter-area traffic transits through area 0.

## Part 3: The Complete Map — Which LSAs Are Allowed Where

| LSA Type | Normal | Stub | Totally Stubby | NSSA | Totally NSSA |
|:--------:|:------:|:----:|:--------------:|:----:|:------------:|
| **1 (Router)** | ✅ | ✅ | ✅ | ✅ | ✅ |
| **2 (Network)** | ✅ | ✅ | ✅ | ✅ | ✅ |
| **3 (Summary)** | ✅ | ✅ | ❌ | ✅ | ❌ |
| **4 (ASBR Summary)** | ✅ | ❌ | ❌ | ❌ | ❌ |
| **5 (External)** | ✅ | ❌ | ❌ | ❌ | ❌ |
| **7 (NSSA External)** | ❌ | ❌ | ❌ | ✅ | ✅ |
| **Default route injected** | No | Yes | Yes | No* | Yes |

*\*NSSA does not inject a default route by default. Use `area X nssa default-information-originate` on the ABR if needed.*

**How to read this table:** Type 1 and 2 are never blocked (they describe the local area — without them, OSPF can't function). Type 3 is blocked only in "totally" variants. Types 4 and 5 are blocked by all stub/NSSA variants. Type 7 only exists in NSSA/Totally NSSA.

## Part 4: How LSA Types Connect to Router Roles

```
Every Router          DR only          ABR              ASBR (inside NSSA)    ASBR (normal area)
generates             generates        generates        generates             generates
Type 1 ──────────── Type 2 ──────── Type 3 + 4 ──────  Type 7 ──────────── Type 5
(Router LSA)        (Network LSA)   (Summary +         (NSSA External)     (AS External)
                                     ASBR Summary)           |
stays in area       stays in area   crosses areas        ABR converts
                                                         Type 7 → Type 5
                                                         at area boundary
```

## Part 5: Key Rules from Class Notes

**Route summarization** can only be done on **ABR** (inter-area, `area X range`) and **ASBR** (external, `summary-address`). Internal routers cannot summarize.

**Redistribution** puts routes from one routing source into another. By default, OSPF redistribution uses **metric-type 2 (E2)** — the metric does not change as the route crosses internal OSPF routers. In my Lab 07: `redistribute ospf 100 metric 10000 metric-type 2 subnets`.

**Virtual-link** is needed when an area is **not directly connected to the backbone (area 0).** It creates a logical extension of area 0 through a transit area. In my Lab 07: area 108 can't reach area 0 directly, so virtual-links chain through area 48.

**LSAs stay in the database** (`show ip ospf database`) even if they aren't in the routing table. The LSDB holds the raw data; the routing table holds the SPF calculation results. A route might be in the LSDB but lose to a better path and not appear in `show ip route`.

## Part 6: Seeing It In my Lab 07

```
! See all LSA types in area 0:
R1# show ip ospf database

! Type 1 (every router in area 0):
R1# show ip ospf database router

! Type 2 (DR-generated, backbone broadcast segment):
R1# show ip ospf database network

! Type 3 (summaries from ABRs R2, R3, R4):
R1# show ip ospf database summary

! Type 4 (ASBR pointers — R1 is ASBR, so other areas get this):
R2# show ip ospf database asbr-summary

! Type 5 (redistributed externals from R1):
R2# show ip ospf database external

! Type 7 (NSSA externals from R5, visible only inside area 35):
R5# show ip ospf database nssa-external

! Compare LSDB to routing table:
R1# show ip ospf database | count       ← total LSAs in LSDB
R1# show ip route ospf | count          ← only the winners made it here
```
